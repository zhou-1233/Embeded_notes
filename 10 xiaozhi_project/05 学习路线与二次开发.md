# 学习路线与二次开发方向

## 目标

用 1 个月（大二升大三暑假）尽量吃透这个项目，为后续智能手表项目和找实习打基础。

---

## 一、学习路线（4 阶段 × 1 周）

### 阶段 1：跑起来（1-2 天）

- [ ] 搭建 ESP-IDF 开发环境
- [ ] 编译烧录，确保能在 CX-ESP32S3 上运行
- [ ] 连接 WiFi，体验完整语音对话流程
- [ ] 用串口监视器看日志输出，理解启动顺序

### 阶段 2：逆向追踪（1 周）

按这条关键链路逆序追踪：

```
main.cc: app_main()
    → Application::Start()          # 看启动流程
    → Application::MainEventLoop()  # 看事件驱动模型
        → AudioService::Initialize()     # 音频初始化
        → Protocol::Start()              # 网络连接
        → AudioService 音频采集 → 编码 → 发送
        → 接收 → 解码 → 播放
        → MCP 消息处理
```

**方法**：每遇到一个不认识的函数/概念，深挖到底。比如：
- 看到 `opus_encoder` → 查 OPUS 编解码原理
- 看到 `afe_audio_processor` → 研究回声消除算法
- 看到 `FreeRTOS event_group` → 查 FreeRTOS 事件组文档

### 阶段 3：模块拆解（2-3 周）

按优先级逐个吃透：

| 优先级 | 模块 | 核心知识点 | 建议时间 |
|---|---|---|---|
| ⭐⭐⭐ | `application.cc` | FreeRTOS 事件循环、状态机设计 | 2 天 |
| ⭐⭐⭐ | `audio_service.cc` | 多任务音频管线、生产者-消费者模式 | 3 天 |
| ⭐⭐⭐ | `protocols/` | WebSocket/MQTT、Observer 模式 | 2 天 |
| ⭐⭐⭐ | `mcp_server.cc` | MCP 协议、JSON-RPC 设计 | 2 天 |
| ⭐⭐ | `audio/codecs/` | I2S 音频驱动、芯片手册阅读 | 3 天 |
| ⭐⭐ | `audio/processors/` | AFE 前端处理、VAD/AEC 原理 | 2 天 |
| ⭐⭐ | `display/` + `lvgl_display/` | LVGL 框架、SPI 屏幕驱动 | 2 天 |
| ⭐⭐ | `boards/common/` | 硬件抽象层设计、代码复用 | 1 天 |
| ⭐ | `boards/cx-esp32s3/` | 板级适配全流程 | 1 天 |
| ⭐ | `ota.cc` | OTA 升级原理、分区表 | 1 天 |

### 阶段 4：融会贯通（1 周）

- [ ] **画一张完整的架构图**（自己画，不抄现成的）
- [ ] **用一句话解释每个文件做什么**（费曼学习法）
- [ ] **回答这些灵魂拷问**：

1. 为什么用事件组（Event Group）而不用信号量（Semaphore）？
2. 为什么音频采集和 OPUS 编码要分开两个任务？
3. MCP 协议和传统 REST API 有什么区别？为什么 LLM 要用 MCP？
4. 93 个开发板怎么做到一套代码兼容的？用了什么设计模式？
5. I2S 和 I2C 虽然都叫 I2，但协议层面有什么本质区别？
6. AFE（Audio Front-End）做了哪三件事？为什么在嵌入式端做？
7. 队列中为什么主要存 OPUS 包而不是 PCM 数据？

## 二、吃透的标准

很多人以为"跑通了"就算吃透了——**那叫用过，不叫吃透。**

### 真正的吃透标准

> 关掉所有代码，能**画出一张架构图**，并且**口头讲清楚**：
> - 音频数据怎么从麦克风流到服务器的
> - 中间经过了几层处理
> - 每一层的 buffer 多大，为什么是这个大小
> - 如果某个环节慢了，会出现什么现象

### 面试级别的检验

如果你能流畅回答以下问题，说明真正吃透了：

1. "这个项目的音频管线延迟大概多少？瓶颈在哪里？"
2. "如果网络抖动导致丢包，你的系统怎么处理？"
3. "怎么做到唤醒词检测的？误唤醒率大概多少？"
4. "MCP Tool 调用是同步的还是异步的？如果 Tool 执行时间很长怎么办？"
5. "如果让你把这个项目移植到一块全新的开发板上，你从哪开始？"

## 三、二次开发方向

普通地"加个传感器"不算有区分度的二次开发。以下按面试加分程度排序：

### 🥇 金奖：端侧推理

| 方向 | 具体做法 | 面试亮点 |
|---|---|---|
| 自训练唤醒词 | 用 Edge Impulse 或 ESP-DL 训练自己的唤醒词模型 | "我在端侧部署了一个深度学习模型" |
| 本地意图识别 | 用小模型（如 SenseVoice）做本地命令词识别 | "我实现了离线 NLU" |
| 异常声音检测 | 用 DSP + 简单 ML 检测婴儿哭声、玻璃破碎等 | "音频事件检测，端侧推理延迟 < 100ms" |

### 🥈 银奖：多模态交互

| 方向 | 具体做法 | 面试亮点 |
|---|---|---|
| MCP 图片 Tool | 接 OV2640 摄像头，MCP Tool 拍照返图给 LLM | "我实现了 MCP 协议的多模态交互" |
| 传感器 MCP Tool 集 | 温湿度、光线、运动传感器都注册为 MCP Tool | "LLM 可以通过 MCP 直接获取环境数据" |
| 显示交互升级 | LLM 返回表情/图片给 LVGL 渲染 | "端云协同的多模态 UI" |

### 🥉 铜奖：跨平台/系统

| 方向 | 具体做法 | 面试亮点 |
|---|---|---|
| 跨芯片移植 | 适配到 ESP32-P4/C5/C6 | "我做了跨芯片平台的移植，理解芯片差异" |
| Linux 迁移 | 部分功能移植到嵌入式 Linux | "RTOS 到 Linux 的系统迁移" |
| 驱动抽象层 | 提取一套通用的外设驱动框架 | "我设计了跨平台的硬件抽象层" |

## 四、工具链与调试技巧

### 开发环境
- **IDE**：VS Code + ESP-IDF 插件
- **调试**：串口日志（ESP_LOG）+ 逻辑分析仪（I2S/I2C 时序）
- **性能分析**：FreeRTOS 任务栈水位监控、heap 内存监控

### 常用调试命令
```bash
# 编译并烧录，指定目标板为 cx-esp32s3
idf.py set-target esp32s3
idf.py build
idf.py -p COMx flash monitor

# 只配置不编译
idf.py menuconfig

# 查看分区表
idf.py partition-table

# 监控各任务栈使用情况
# 在 sdkconfig 中开启:
# CONFIG_FREERTOS_GENERATE_RUN_TIME_STATS=y
# CONFIG_FREERTOS_USE_TRACE_FACILITY=y
```
