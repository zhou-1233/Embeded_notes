### 1.顶层视角
操作所有功能，记录每种输入对应的输出，知道这个系统到底在干什么

### 2.吃透硬件：
**从原理图入手，看懂板子:** MUC主控，电源树，关键外设及其占用哪些引脚。
**划分功能模块：** 哪些是电源管理区，哪些是传感器采集区，硬件连接决定了软件的数据流向
**怎么算吃透：** 能脱离原理图，手绘硬件框图，可替换元器件，自主修改硬件和配套驱动代码

### 3.梳理工程目录：
**先看骨架：** 梳理工程框架，每层文件干什么，每个文件干什么。
**再看流程：** 从main函数切入，顺着代码理清楚初始化流程。
**用数据流驱动代码阅读** ，而不是逐行死磕
**输入流：** 传感器数据->驱动层采集（I2C/ADC）->滤波/校准算法->应用逻辑
**输出流：** 控制决策->PWM输出/GPIO翻转->电机/灯光
**通信流：** 接收到的串口/网络包->协议解析->命令执行->结果封装打包
**画图：** 边看代码边画“数据流图”或“任务间通信图”（如队列、信号量），这是理解多任务系统的核心

**怎么算吃透：** 能解释每个模块的作用，手绘主函数执行流程，RTOS任务调度逻辑，数据流转图，

### 4.分层逐段啃代码：
1. 底层初始化：对照芯片手册，看懂时钟，引脚，中断配置
2. 外设驱动：把单个外设从工程剥离，新建空白工程单独编译测试，能独立跑通代表看懂驱动
3. RTOS中间件：梳理每个任务的功能，任务优先级，队列/信号量数据流向
4. 应用业务：手绘业务流程图

### 5.动手修改验证
1. 小改测试，修改IO引脚，采集周期，上报频率，下载验证现象
2. 二次开发：原有项目新增功能（如原有串口上报，新增WiFi云平台上传），找出缺陷并落地优化
3. 精简项目：删除冗余无用的代码，工程正常运行

### 四、怎么吃透这个项目（学习路线）

#### 阶段 1：跑起来（1-2天）

  1. 编译烧录，确保能在你的 CX-ESP32S3 上运行
  2. 连接 WiFi，体验完整语音对话流程
  3. 用串口监视器看日志输出，理解每个模块的启动顺序

#### 阶段 2：逆向追踪（1周）

  **按这条链路逆序追踪（这是关键方法）：** 

  main.cc: app_main()
      → Application::Start()          # 看启动流程
      → Application::MainEventLoop()  # 看事件驱动模型
          → AudioService::Initialize()    # 音频初始化
          → Protocol::Start()             # 网络连接
          → AudioService 音频采集 → 编码 → 发送
          → 接收 → 解码 → 播放
          → MCP 消息处理

  **每遇到一个不认识的函数/概念，深挖到底——比如看到 opus_encoder，就去查 OPUS 的原理；看到
  afe_audio_processor，就去研究回声消除算法。**
