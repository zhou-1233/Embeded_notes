## 核心设计模式

| 模式 | 实现位置 | 说明 |
|---|---|---|
| **单例模式** | `Application`, `AudioService`, `McpServer` | 全局唯一实例 |
| **策略模式** | `Protocol` → `WebsocketProtocol` / `MqttProtocol` | 运行时切换通信协议 |
| **观察者模式** | `AudioServiceCallbacks`, Protocol 回调 | 事件驱动 |
| **生产者-消费者** | `audio_encode_queue_`, `audio_decode_queue_` | FreeRTOS 队列 + 条件变量 |
| **模板方法** | `AudioCodec` → 各芯片驱动 | 统一接口，不同实现 |
| **RAII** | `TaskPriorityReset` | 任务优先级临时调整（v2.0.4 新增） |

## AudioService 内部架构

```
AudioService
│
├── 音频输入任务 (AudioInputTask)
│   └── 从 Codec 读 PCM → 喂给 Processor → 创建 AudioTask → 入编码队列
│
├── OPUS 编解码任务 (OpusCodecTask)
│   ├── 从编码队列取 PCM → OPUS 编码 → 入发送队列
│   └── 从解码队列取 OPUS → OPUS 解码 → 入播放队列
│
├── 音频输出任务 (AudioOutputTask)
│   └── 从播放队列取 PCM → 写到 Codec 输出
│
├── 编码队列 (audio_encode_queue_)
├── 发送队列 (audio_send_queue_)      ← 上行，给 Protocol
├── 解码队列 (audio_decode_queue_)    ← 下行，来自 Protocol
└── 播放队列 (audio_playback_queue_)
```

### 为什么音频采集和 OPUS 编码要分开两个任务？

这是面试高频问题。原因：

1. **解耦**：Codec 读取是 I2S 中断驱动的，OPUS 编码是 CPU 密集型。放一起会互相阻塞
2. **缓冲区管理**：采集端可能需要累积到 60ms 帧长再编码
3. **弹性**：编码队列有 2 个槽位（`MAX_ENCODE_TASKS_IN_QUEUE = 2`），允许采集比编码快一点，不会丢数据

### 队列设计原则

```
音频数据量：PCM 16bit × 24000Hz = 384kbps ≈ 48KB/s
OPUS 压缩后：~16-32kbps ≈ 2-4KB/s

所以：
- "编码/解码队列"存 OPUS 包（小）→ 可以缓存很多（40 个）
- "发送/播放队列"存 OPUS 包（小）→ 同上
- "编码任务队列"存 PCM 数据（大）→ 只缓存 2 个（内存限制）
```

## MCP Server 架构

```cpp
// mcp_server.h 核心类
class McpServer {
    // MCP 协议核心方法
    void ParseMessage(const cJSON* json);  // 解析 JSON-RPC 消息
    void GetToolsList(int id, ...);         // 返回可用工具列表
    void DoToolCall(int id, ...);           // 执行工具调用

    std::vector<McpTool*> tools_;           // 注册的工具列表
};

class McpTool {
    std::string name_;              // 工具名称
    std::string description_;       // 工具描述
    PropertyList properties_;       // 参数定义
    std::function<ReturnValue(...)> callback_;  // 实际回调函数
    bool user_only_;               // 是否仅用户可见（不对 AI 暴露）
};

// v2.0.4 新增返回类型
using ReturnValue = std::variant<
    bool,           // 布尔
    int,            // 整数
    std::string,    // 字符串
    cJSON*,         // 结构化 JSON
    ImageContent*   // 图片（多模态）
>;

class ImageContent {   // v2.0.4 新增
    std::string encoded_data_;   // Base64 编码的图片数据
    std::string mime_type_;      // 图片 MIME 类型
};
```

## 93 个开发板的兼容机制

```
boards/
├── common/
│   ├── board.h          ← 抽象基类 Board
│   ├── wifi_board.h     ← WiFi 板
│   ├── ml307_board.h    ← 4G 板
│   └── ...
│
├── cx-esp32s3/
│   ├── config.h         ← #define 引脚映射（编译时选择）
│   └── cx_esp32s3.cc    ← 具体板级实现
│       class CxEsp32s3Board : public WifiBoard {
│           // 重写 InitializeAudioCodec()
│           // 重写 InitializeLcdDisplay()
│           // 重写 InitializeLed()
│           // 注册 MCP Tools
│       }
```

核心模式：**编译时多态**
- `config.h` 用宏定义引脚（不占 RAM），编译时就确定了
- 每个板子继承公共基类，重写虚函数
- `config.json` 控制 sdkconfig 参数，选择编译哪个板子
- 一套代码，通过编译选项切换目标板
